# R2 Blob Storage for Asset Uploads

**Date**: 2026-04-06
**Status**: Implemented
**Author**: AI-assisted

## Overview

Add R2 blob storage to `apps/api` so paid users can upload assets (images, PDFs) that get embedded in workspace documents. Free users reference external URLs instead.

```
┌─────────────────────────────────────────────────────────────────┐
│  THE SETUP                                                       │
│                                                                  │
│  R2 bucket: PRIVATE (no public domain, no r2.dev)                │
│  Only reachable via Worker binding (env.ASSETS_BUCKET)           │
│  Internet ──✕──→ R2    (impossible)                              │
│  Internet ──→ Worker ──→ R2    (the only path)                   │
└─────────────────────────────────────────────────────────────────┘
```

## Motivation

### Current State

The API has no file storage. All bindings are for structured data:

```
SESSION_KV     → Better Auth session cache
WORKSPACE_ROOM → Durable Object (Yjs sync)
DOCUMENT_ROOM  → Durable Object (Yjs sync)
HYPERDRIVE     → Postgres (auth tables, DO tracking)
ASSETS         → Static (Dashboard SPA)
```

Users have no way to embed uploaded images or PDFs in their documents through Epicenter's infrastructure.

### Problems

1. **No asset hosting**: Users who want images in documents must host them externally and paste URLs
2. **No upload gating**: No infrastructure to gate uploads behind paid plans
3. **No storage accounting**: No way to track per-user storage for future billing

### Desired State

Paid users upload files via `POST /api/assets`. Files are stored in R2. Documents reference assets via `GET /api/assets/:id`. Storage can later be tracked via Autumn's non-consumable metered features.

## Research Findings

### R2 Access Control

R2 access control is **bucket-level only**—no per-object ACLs. A private R2 bucket (no public custom domain, no `r2.dev` subdomain) is completely inaccessible from the internet. The ONLY access path is the Worker binding (`env.ASSETS_BUCKET`). Cloudflare's own docs confirm: "You must define authorization logic to determine who can perform what actions to your bucket. This logic lives within your Worker's code."

Options investigated:

| Approach | How It Works | Tradeoff |
|---|---|---|
| Public bucket + custom domain | All objects publicly readable | No per-object control, need WAF rules for restrictions |
| Public bucket + WAF rules | WAF restricts paths on the custom domain | Fragile at scale, bucket-level only |
| Private bucket + Worker proxy | All reads go through Worker, Worker decides access | Full control, security headers, future auth possible |

**Key finding**: Private bucket + Worker-proxied reads is the standard production pattern. It gives full control over response headers, future ability to add per-request auth, and no public domain to manage.

### Read Authentication and `<img>` Tags

The `<img>` tag auth problem is decisive for the public-vs-private reads question. Browsers don't send `Authorization` headers on `<img>` requests. The Tauri desktop app's webview origin (`tauri://localhost`) doesn't share cookies with `api.epicenter.so`.

Solutions investigated for authenticated reads:

| Solution | Complexity | UX Impact |
|---|---|---|
| JS fetch → blob URL | Every image loads via JS, converted to blob URL | Visible loading delay, no native caching |
| Service Worker intercept | SW intercepts requests, injects auth | Doesn't work in Tauri webview |
| Short-lived signed URLs | Server mints a URL per image per render | Extra round-trip per image, cache-busting |
| Unauthenticated + unguessable IDs | No auth on reads, random ID is the access control | `<img>` tags just work everywhere |

**Key finding**: Unauthenticated reads with unguessable IDs is the industry standard (Supabase Storage, Firebase Storage, Cloudinary, Discord). Asset IDs are 15-char nanoids (36^15 ≈ 2.2 × 10²³)—computationally infeasible to brute-force with no enumeration endpoint.

### Security Model

```
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 1: Private bucket                                         │
│  No public domain. No r2.dev. The internet can't reach R2.       │
├─────────────────────────────────────────────────────────────────┤
│  LAYER 2: Worker gatekeeper                                      │
│  All reads proxied. Worker sets security headers, controls       │
│  everything. Auth can be added later without changing URLs.      │
├─────────────────────────────────────────────────────────────────┤
│  LAYER 3: Unguessable URLs                                       │
│  /api/assets/{userId}/{assetId}                                  │
│  Both are 15-char nanoids from [a-z0-9]                          │
│  Combined keyspace: 36³⁰ ≈ 4.9 × 10⁴⁶                          │
│  At 1B guesses/sec with known userId → ~700,000 years            │
├─────────────────────────────────────────────────────────────────┤
│  LAYER 4: No enumeration                                         │
│  No endpoint lists assets. No way to discover valid IDs.         │
├─────────────────────────────────────────────────────────────────┤
│  LAYER 5: Where do URLs live?                                    │
│  Only inside encrypted CRDT documents. To get the URL,           │
│  you'd need to decrypt the document—at which point you           │
│  already have the data.                                          │
└─────────────────────────────────────────────────────────────────┘
```

Same model as Google Drive "anyone with the link," Discord CDN, Slack files, presigned S3 URLs.

### ID Generation

The codebase already has ID generation in `packages/workspace/src/shared/id.ts`:

```ts
const ALPHABET = 'abcdefghijklmnopqrstuvwxyz0123456789';
const nanoid10 = customAlphabet(ALPHABET, 10);  // generateId() — table-scoped
const nanoid15 = customAlphabet(ALPHABET, 15);  // generateGuid() — globally unique
```

`generateGuid()` produces 15-char lowercase alphanumeric strings. URL-safe, no encoding needed, sufficient collision resistance for millions of objects.

### Autumn Storage Billing

Autumn models storage as a **non-consumable metered feature**. `track()` takes deltas (positive = deduct, negative = credit). `usage()` sets absolute totals. `check()` evaluates remaining balance + overage.

Current `autumn.config.ts` has only `aiUsage` and `aiCredits`—no storage feature exists yet. The phase 4 storage billing spec planned a `storage-bytes` feature but it was never implemented.

**Implication**: Storage metering is additive work that pairs with phase 4 DO storage billing. Not needed for v1 upload gating—a simple `planId !== 'free'` check suffices.

### Cloudflare R2 Billing (Account-Level Only)

Cloudflare bills the account for total R2 usage—there is no per-prefix or per-user breakdown:

| Cost | Amount | Free Tier |
|---|---|---|
| Storage | $0.015 / GB-month | 10 GB/month |
| Class A ops (put, list, copy) | $4.50 / million | 1M/month |
| Class B ops (get, head) | $0.36 / million | 10M/month |
| Delete | Free | — |
| Egress | Free | — |

There is no Cloudflare API for per-user storage totals, no event notifications for upload/delete, and no mechanism to attribute storage to individual users. Per-user storage billing requires application-level tracking—which is what Autumn's non-consumable metered feature provides.

### R2 customMetadata

Production-ready, GA feature. Part of the core Workers API with no beta tags.

| Property | Type | Limit |
|---|---|---|
| `customMetadata` | `Record<string, string>` | 8,192 bytes total (shared with httpMetadata) |
| `httpMetadata` | `R2HTTPMetadata` | Included in the 8KB limit |
| Object key length | string | 1,024 bytes |

Our customMetadata (~140 bytes for originalName + userId + uploadedAt) uses <2% of the 8KB limit.

### R2 Features We Leverage

```
┌────────────────────────────┬────────────────────────────────────┐
│  Feature                   │  What We Get                       │
├────────────────────────────┼────────────────────────────────────┤
│  writeHttpMetadata()       │  One call sets Content-Type etc.   │
│  ETags + If-None-Match     │  304 Not Modified → fast caching   │

└────────────────────────────┴────────────────────────────────────┘
```

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Upload flow | Direct (client → Worker → R2) | Simpler than presigned URLs. 25MB limit is well within Workers' 100MB body limit. Auth stays in one place. |
| Read access | Unauthenticated, unguessable IDs | `<img>` tags can't carry auth headers. 15-char nanoid keyspace is computationally infeasible to brute-force. Industry standard pattern. |
| R2 key structure | `{userId}/{assetId}` | User prefix enables `list({ prefix })` for per-user operations. Both IDs are opaque nanoids—standard practice (S3 uses bucket names, GitHub uses usernames, Discord uses channel IDs, Supabase uses project refs in URLs). |
| ID generation | `generateGuid()` (existing 15-char nanoid) | Already in codebase, URL-safe, globally unique, sufficient entropy. No new dependency. |
| File extension in key | No | MIME type stored in R2 `httpMetadata.contentType`. Original filename in `customMetadata.originalName`. Avoids collisions and encoding issues. |
| SVG support | No | SVG is a stored XSS vector (`<script>` tags). Not worth the attack surface for "images in documents." Users can host SVGs externally. |
| Upload gating | `planId !== 'free'` | Simple boolean check. Matches existing AI plan-gating pattern. No Autumn storage feature needed for v1. |
| Storage accounting | Deferred to phase 4 | Storage metering requires adding an Autumn feature + items to all plans + `atmn push`. Clean additive change when phase 4 DO storage billing is implemented. |
| Bucket visibility | Private (no custom domain) | Reads go through Worker for security headers, Content-Disposition, and future auth capability. |
| Delete authorization | Owner only | `c.var.user.id` must match asset's userId prefix. |

## Architecture

### Upload Flow

```
┌─────────────────────────────────────────────────────────────────┐
│  UPLOAD  (POST /api/assets)                                      │
│                                                                  │
│  Client ──[auth + file]──→ Worker                                │
│                              ├─ authGuard: logged in?            │
│                              ├─ planCheck: paid plan?            │
│                              ├─ validate: MIME in allowlist?     │
│                              ├─ validate: size ≤ 25MB?           │
│                              │                                   │
│                              ├─ assetId = generateGuid()         │
│                              ├─ key = "{userId}/{assetId}"       │
│                              └─ ASSETS_BUCKET.put(key, file, {   │
│                                   httpMetadata: { contentType }, │
│                                   customMetadata: {              │
│                                     originalName, userId,        │
│                                     uploadedAt                   │
│                                   }                              │
│                                 })                               │
│                                                                  │
│  Returns: { id, url, contentType, size, originalName }           │
└─────────────────────────────────────────────────────────────────┘
```

### Read Flow

```
┌─────────────────────────────────────────────────────────────────┐
│  READ  (GET /api/assets/:userId/:assetId)                        │
│                                                                  │
│  <img src="https://api.epicenter.so/api/assets/k7x.../r5t...">  │
│    │                                                             │
│    └──→ Worker                                                   │
│           ├─ NO auth check (URL is the credential)               │
│           ├─ key = "{userId}/{assetId}"                          │
│           ├─ object = ASSETS_BUCKET.get(key, {                   │
│           │    onlyIf: request.headers,  ← ETag → 304           │
│           │    range: request.headers    ← partial → 206         │
│           │  })                                                  │
│           ├─ object.writeHttpMetadata(headers)                   │
│           └─ adds: ETag, nosniff, Content-Disposition            │
│                                                                  │
│  Returns: file bytes with security headers                       │
│  (or 304 Not Modified if ETag matches)                           │
└─────────────────────────────────────────────────────────────────┘
```

### R2 Object Structure

```
Key:            {userId}/{assetId}
                e.g. "k7x9m2p4q8abc12/r5t8w3z6y1bcd09"

httpMetadata:
  contentType:  "image/png"         (validated server-side)

customMetadata:
  originalName: "vacation-photo.png" (for Content-Disposition)
  userId:       "k7x9m2p4q8abc12"   (redundant with key prefix, useful for queries)
  uploadedAt:   "2026-04-06T..."     (ISO timestamp)
```

### Key Lookup for Reads

```
┌──────────────────────────────────────────────────────────────────┐
│  R2 key: "{userId}/{assetId}"                                     │
│  URL:    /api/assets/{userId}/{assetId}                           │
│                                                                   │
│  Without userId in the URL, the Worker can't construct the key.   │
│                                                                   │
│  Option A: Scan R2 ────→ list() is paginated, no suffix search   │
│  Option B: KV lookup ──→ extra service, eventual consistency     │
│  Option C: userId in URL → one R2 get(), zero extra services  ✓  │
│                                                                   │
│  The userId is an opaque 15-char nanoid. It reveals nothing.      │
│  S3 has bucket names in URLs. GitHub has usernames. Discord       │
│  has channel IDs. Supabase has project refs. This is standard.    │
└──────────────────────────────────────────────────────────────────┘
```

## Edge Cases

### Upload with Invalid MIME Type

1. User uploads a `.exe` renamed to `.png`
2. Server validates `Content-Type` header from multipart form against allowlist
3. Returns 415 Unsupported Media Type with `{ error: 'File type not allowed', allowed: [...] }`
4. Even if a mistyped file were stored, the read handler sets `Content-Type` from R2 httpMetadata + `X-Content-Type-Options: nosniff`, so browsers won't execute it. Magic byte validation is unnecessary—skip it.

### Upload Exceeds Size Limit

1. User uploads a 30MB file
2. Hono `bodyLimit` middleware rejects before the handler runs
3. Returns 413 with standard error

### Delete Non-Existent Asset

1. User deletes an asset that was already deleted
2. R2 `delete()` is idempotent—no error
3. Return 204 regardless

### Delete Asset Owned by Another User

1. User A tries `DELETE /api/assets/{userB_id}/{assetId}`
2. Owner check: `c.var.user.id !== userId` → 403 Forbidden

### Concurrent Uploads Near Storage Quota (Phase 4)

1. User at 4.9GB of 5GB quota uploads two 200MB files simultaneously
2. Both pass `autumn.check()` before either is tracked
3. Total goes to 5.3GB—slight overage
4. **Acceptable**: Autumn's cumulative metering tolerates minor overages between syncs. Periodic `usage()` reconciliation corrects drift.

## Open Questions

1. **Per-upload size limit**: Is 25MB right?
   - Workers allow 100MB+ but we buffer the full body
   - 25MB is generous for images/PDFs in documents
   - **Recommendation**: Start at 25MB, adjust based on usage.

## Review

**Completed**: 2026-04-06
**Branch**: feat/fix-dashboard

### Summary

Implemented v1 R2 blob storage with upload, read, and delete endpoints. Private R2 bucket with Worker-proxied reads. Unguessable nanoid URLs as the credential for reads. Plan gating via Autumn customer lookup (planId !== 'free').

### Deviations from Spec

- **Routing split**: Used two separate Hono sub-routers (`assetAuthedRoutes` and `assetPublicRoutes`) instead of a single route group. This cleanly separates authenticated (upload/delete) from unauthenticated (read) routes without complex middleware ordering.
- **Precondition check**: Used `'body' in object` to distinguish `R2ObjectBody` from `R2Object` per CF typed API, rather than checking `object.body` directly.
- **Fixed pre-existing type error**: Regenerating `worker-configuration.d.ts` exposed a stale `as` cast on the ASSETS binding in dashboard SPA routes. Fixed by using `c.env.ASSETS` directly since the binding is now properly typed.
