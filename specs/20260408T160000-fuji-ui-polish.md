# Fuji UI Polish

**Date**: 2026-04-08
**Status**: Implemented
**Author**: AI-assisted

## Overview

Six targeted UI improvements to make Fuji feel like a finished desktop app rather than a functional prototype. Each change uses existing `packages/ui` components and follows patterns already proven in `apps/opensidian`.

## Motivation

### Current State

Fuji's layout is a `SidebarProvider` wrapping a sidebar + main area. There's no persistent header bar, no global status bar, no resizable panels, and empty states use inline text instead of the shared `Empty` component. The view-mode toggle is a lone icon button floating in the top-right corner with no surrounding context.

```
┌─────────────────────────────────────────────────────┐
│ Sidebar        │  [view toggle icon]                │
│ ┌─ Fuji ─────┐ │  ┌─────────────────────────────┐  │
│ │ Search...  │ │  │ Timeline            [+]     │  │
│ │ All (0)    │ │  │                             │  │
│ │            │ │  │                             │  │
│ │            │ │  │    No entries yet.           │  │
│ │            │ │  │    + Create your first entry │  │
│ │            │ │  │                             │  │
│ └────────────┘ │  └─────────────────────────────┘  │
│ [rail]         │                                    │
└─────────────────────────────────────────────────────┘
```

Problems:

1. **No app identity in the main area**—the "Fuji" label is inside the sidebar; collapse it and the branding disappears.
2. **No external links or meta actions**—GitHub, theme toggle, and settings are unreachable without the command palette.
3. **The "+" create button moves**—it's inside the table/timeline header, but disappears when viewing the editor.
4. **Empty states are plain text**—the `Empty` compound component exists in `packages/ui` but isn't used.
5. **Sidebar width is fixed**—`SidebarProvider` supports collapse via rail but not drag-to-resize.
6. **No global status bar**—`StatusBar` only appears inside the editor; browsing entries shows no contextual info.

## Research Findings

### Existing Patterns in the Monorepo

| Pattern | Opensidian Implementation | Fuji Current |
|---|---|---|
| App header with branding + action icons | `SidebarHeader.svelte` — logo, app name, search/new-folder/new-file buttons with tooltips | `FujiSidebar.svelte` header — "Fuji" text + search input only |
| Resizable panels | `AppShell.svelte` — `Resizable.PaneGroup` + `Resizable.Handle withHandle` for sidebar/editor/terminal/chat | `SidebarProvider` — collapse only, no drag resize |
| Global status bar | `editor/StatusBar.svelte` — line/col, word count, Terminal/AI Chat toggles, VIM indicator, settings gear with LightSwitch | `StatusBar.svelte` — editor-only, word count + timestamps |
| Empty states | Not applicable (file tree always shows) | Inline `<p>` text in table/timeline |
| Theme toggle | `LightSwitch` inside StatusBar settings popover | `ModeWatcher` in layout, no UI toggle |
| External links | GitHub icon in header (visible in opensidian.com UI) | None |

**Key finding**: Opensidian already solved every pattern Fuji needs. The `apps/opensidian` codebase is the reference implementation.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| App header placement | Above everything (outside sidebar + main) | Matches Opensidian's `SidebarHeader` pattern; branding stays visible when sidebar collapses |
| Sidebar resize approach | Replace `SidebarProvider` with `Resizable.PaneGroup` | `SidebarProvider` doesn't support drag resize; Opensidian already proves the Resizable approach works |
| Create button location | Both in app header AND in table/timeline headers | Header provides always-visible access; sub-headers provide contextual access. Opensidian does the same (new file in sidebar header + right-click in tree) |
| Status bar scope | Global (always visible) | Opensidian's StatusBar is always present regardless of what panel is focused |
| Empty state component | `Empty.*` from `@epicenter/ui/empty` | Already exists, already documented in the svelte skill's loading/empty state patterns, currently unused in Fuji |
| Keep rail collapse | Remove | Resizable handle replaces the rail's purpose; having both would be confusing |

## Edge Cases

### Sidebar at Minimum Width

1. User drags sidebar to `minSize` (15%).
2. Search input may truncate — ensure it uses `w-full` and truncates gracefully.
3. Entry counts in sidebar groups may wrap — they use `ml-auto text-xs` which should truncate.

### Mobile / Narrow Viewport

1. Resizable panels don't work well on mobile.
2. **Recommendation**: Add a `min-w-[768px]` or `@media` breakpoint that hides the sidebar on small screens. Defer this to a follow-up.

### Editor Status Bar vs Global Status Bar

1. When the editor is open, both the editor's `StatusBar` and the `GlobalStatusBar` are visible.
2. This is intentional — Opensidian does the same (editor status bar shows line/col/words, global bar shows Terminal/Chat/Settings).
3. The editor StatusBar shows entry-specific info (word count, dates). The global StatusBar shows workspace-level info (entry count, shortcuts).

### Command Palette Trigger

1. `AppHeader` has a search button that opens the command palette.
2. The `⌘K` keyboard shortcut still works (handled in `svelte:window onkeydown`).
3. Both paths set `paletteOpen = true` — no conflict.

## Open Questions

1. **Should the sidebar remember its width?**
   - Opensidian doesn't persist pane sizes (resets on reload).
   - Options: (a) Don't persist (simplest), (b) Persist to localStorage via `onResize` callback.
   - **Recommendation**: Don't persist for now. Add later if users request it.

2. **Should the global status bar show sync status?**
   - Fuji's workspace has sync extensions but we'd need to expose connection state.
   - Options: (a) Show just entry count for now, (b) Add sync indicator.
   - **Recommendation**: Start with entry count only. Sync indicator is a follow-up.

3. **Should the view-mode toggle move to the app header?**
   - Currently it's in its own thin header row above the table/timeline.
   - Options: (a) Keep where it is, (b) Move to app header right section, (c) Move to status bar.
   - **Recommendation**: Keep where it is for now. It's contextual to the list view and disappears when editing — that's correct behavior.
